## 为什么要做这个项目

AI 编码代理已经能够生成、重构、测试、Review 和运行远超人类逐行阅读能力的代码量。工程瓶颈因此发生了变化：最难的工作不再是敲代码，而是在实现速度大幅提升之后，仍然控制意图、架构、质量、运行过程和最终责任。

这套方法来自业务系统、Mobile App、机器学习、语音、IoT、底层硬件相关项目、云上部署和 Agent Skill 的真实实践。根据维护者目前的跨项目统计估算，相关实践由 AI 生成超过 30 万行应用代码，维护者没有亲手编写这些代码。该统计口径尚未经过独立审计，因此它是个人实践估算，不是行业基准或普遍效率结论。它说明的是一个结构性变化：当实现动作被高度委托以后，人的主要工作会转向工程控制系统。

## 核心命题

> 编码可以委托，工程责任不能委托。

人负责：

- 问题、价值、范围和非目标；
- 架构、算法方向、产品审美和关键取舍；
- 风险、证据标准、发布权限和最终后果。

AI 负责：

- 研究资料和比较方案；
- 形成实现计划和生成代码；
- 运行测试和工具；
- 分析架构、功能和实现细节；
- 检索日志、Trace、指标和失败工件；
- 执行重复性的开发与测试运维工作。

## 一条完整闭环

```text
意图 → 规格 → 设计 → 方案 → 实现 → Review → 测试 → 观察 → 诊断 → 学习
 ↑                                                                    │
 └────────────────── 更新文档、测试和 Skill ──────────────────────────┘
```

方法由四个平面构成：

| 平面 | 目的 | 主要产物 |
|---|---|---|
| 控制面 | 定义意图、边界、架构、风险和验收 | 需求、NFR、ADR、契约、计划、Skill |
| 执行面 | 让 AI 在批准范围内实现 | 代码、配置、迁移、测试和构建产物 |
| 证据面 | 用证据证明行为，而不是相信流畅输出 | 测试、日志、Trace、指标、基准、Review 报告 |
| 学习面 | 把成功与失败变成长期控制能力 | 复盘、回归测试、模板和版本化 Skill |

## 这个项目与常见内容有什么不同

它不是提示词合集，不是模型排行榜，也不主张让自主 Agent 对生产系统负责。它关注的是以下场景所需要的完整方法：

- 代码量已经不适合人工逐行 Review；
- 架构、审美、复杂算法或物理系统需要人的专业判断；
- 正确性必须通过独立测试和运行证据证明；
- 日志和 Trace 同时服务人和 AI 的诊断；
- 每次缺陷都回写为 Skill、测试和质量门；
- AI 可以高度参与开发测试运维，但生产环境仍然按风险受控。
