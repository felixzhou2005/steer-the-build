# 第五部分 项目案例与公开发布策略
这一部分直接回答两个现实问题：是否应当加入项目介绍，以及怎样把方法发布到 GitHub 和 LinkedIn，让内容可搜索、可贡献，同时避免变成已有“AI 编程最佳实践”的重复版本。

## 37. 是否加入项目介绍：必须加入，但要作为方法证据而不是项目清单
建议加入。没有案例，这套内容容易被看成一组正确但抽象的原则；有了跨业务、移动、语音、IoT、算法和云部署的项目，读者可以看到同一方法如何在不同风险和技术条件下工作。项目介绍的重点不是展示功能多，而是说明“人的决定是什么、AI 执行了什么、哪里失败、用什么证据修正、最终沉淀了什么”。

### 38.1 首批深写五个，其他项目先做短案例
第一版 GitHub 不建议同时把十个项目都写成长文。优先选择能覆盖最大方法跨度的五个：房屋估值算法、iOS App、IoT 语音/远程控制、AI 前台与订单同步、OpenClaw 云上服务。它们分别代表算法、移动、硬件、业务事务和平台运维。其他项目先用一至两页短案例，等有指标、失败证据和可公开材料后再扩写。

> **•** 算法案例突出：人的建模、指标与异常逆向分析。
>
> **•** iOS 案例突出：Figma Skill、Maestro、生命周期与视觉质量。
>
> **•** IoT 案例突出：跨层架构、硬件约束、可观察性与安全失败。
>
> **•** AI 前台突出：语音链路、业务一致性、人工接管与审计。
>
> **•** OpenClaw 服务突出：AI DevOps、Skill 治理、多租户和供应链安全。

## 39. 每个案例的统一写法：让读者能够比较和贡献
| **顺序** | **章节**       | **关键问题**                                                            |
|----------|----------------|-------------------------------------------------------------------------|
| 1        | 背景与成功标准 | 要解决什么，谁使用，成功/失败如何量化？                                 |
| 2        | 完整工程链条   | 需求、设计、架构、Figma/API/数据/算法、计划、测试和运维留下了什么文档？ |
| 3        | 人的关键决策   | 哪些地方不能交给 AI：审美、架构、算法、数据、风险或选型？               |
| 4        | AI 的执行范围  | 研究、方案、代码、测试、Review、日志分析、命令和部署分别做到什么程度？  |
| 5        | 第一次失败     | 哪个结果不符合预期，为什么直接重试无效？                                |
| 6        | 证据与逆向诊断 | 如何拆分结果，找到第一个异常点，用了哪些日志、测试、Trace 或样本？      |
| 7        | 修复与质量门   | 修改了需求/架构/代码/测试/日志中的哪一层？                              |
| 8        | 指标与代价     | 周期、人工时间、测试量、性能、缺陷、成本和后续维护如何？                |
| 9        | Skill 与复用   | 沉淀了哪些基础、领域、项目和任务 Skill？                                |
| 10       | 仍然未知       | 方法的适用边界、未解决风险和下一步实验是什么？                          |

## 45. 公开项目介绍前必须增加的安全、隐私与合规检查
你的项目里有爬取、销售 Lead、语音电话、订单、IoT 控制和 OpenClaw Skill / 服务化，这些比普通代码案例更有吸引力，也更容易触及授权、隐私、安全和平台规则。公开时必须把这些控制写成方法的一部分，不能只展示技术效果。

| **领域**              | **公开前检查**                                                                               |
|-----------------------|----------------------------------------------------------------------------------------------|
| 爬取与 Lead           | 数据是否允许采集、用途是否兼容、是否涉及个人信息、联系是否有合法依据、是否提供退出和删除机制 |
| 自动发帖              | 平台条款、机器人标识、频率、内容审核、版权、误发回滚和账号权限隔离                           |
| 电话翻译 / AI 前台    | 通话告知与同意、录音和转写保留、敏感信息脱敏、金额/订单复核、人工接管和错误免责声明          |
| IoT 控制              | 身份、授权、设备绑定、重放保护、危险指令、失联默认状态、物理急停和完整审计                   |
| ML / 房屋估值         | 数据权利、偏差、地域与时间适用性、置信度、异常样本、人工复核和用途声明                       |
| OpenClaw Skill / AaaS | 第三方 Skill 当作代码审查、租户/容器隔离、最小权限、秘密管理、网络出口、审计、更新和紧急吊销 |
