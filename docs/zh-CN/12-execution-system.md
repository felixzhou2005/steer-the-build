# 第三部分 可执行的 AI 原生工程体系
> *这一部分把方法论变成项目可以直接采用的交付物、质量门、自治级别、指标和类型化检查表。*

## 22. 最小交付物体系
| **编号** | **交付物**        | **最低内容**                                   |
|----------|-------------------|------------------------------------------------|
| A0       | 问题说明书        | 用户、痛点、价值、成功指标、替代方案、停止条件 |
| A1       | 需求规格          | 场景、规则、边界、异常、验收标准、范围外       |
| A2       | 非功能与风险清单  | 性能、安全、可靠性、兼容、成本、可运维性       |
| A3       | 设计与 ADR        | UI、API、模块、数据、算法、部署、方案取舍      |
| A4       | 测试与证据计划    | 不变量、测试层次、样本、基线、阈值、验收人     |
| A5       | 任务卡/代码方案   | 范围、文件、接口、步骤、测试、风险、回滚       |
| A6       | 变更与验证报告    | 实际改动、命令、结果、偏差、已知问题           |
| A7       | 发布与回滚计划    | 迁移、灰度、监控、回滚、负责人、窗口           |
| A8       | 复盘与 Skill 更新 | 根因、行动、回归证据、规则和版本变化           |

小型任务可以把 A1—A5 合并为一张任务卡，但每个字段仍应被回答。大型或高风险项目则应拆分、评审并建立版本。

## 23. 单任务八步闭环
| **步骤**      | **要求**                                         |
|---------------|--------------------------------------------------|
| 1\. Spec      | 明确问题、范围、验收和非目标                     |
| 2\. Plan      | AI 生成代码方案、影响范围、测试和风险            |
| 3\. Critique  | 人或独立审查角色挑战复杂度、遗漏与错误假设       |
| 4\. Implement | AI 在授权范围内执行小批变更                      |
| 5\. Verify    | 运行静态检查、单元、集成、端到端和安全验证       |
| 6\. Observe   | 检查日志、指标、性能和真实流程                   |
| 7\. Simplify  | 删除无用抽象、重复和额外兜底，确认没有范围外改动 |
| 8\. Record    | 更新文档、ADR、回归测试、变更说明和 Skill        |

## 24. 风险分级与 AI 自治等级
不是所有任务都需要同样的人工介入。最有效的方式是根据可逆性、影响范围、数据敏感度和证据强度分配 AI 自治。

| **等级**      | **AI 可做什么**                                  | **典型任务**                       | **人工要求**                         |
|---------------|--------------------------------------------------|------------------------------------|--------------------------------------|
| L0 建议       | 只分析、给选项，不修改代码                       | 架构、算法选择、安全方案           | 人做全部决策                         |
| L1 局部编辑   | 修改限定文件并提供差异                           | 文档、样式、简单 UI、测试补充      | 合并前抽查与自动验证                 |
| L2 受控任务   | 完成单一功能、运行测试、生成报告                 | 普通业务逻辑、API、小型重构        | 先审方案，质量门通过后合并           |
| L3 流水线执行 | 在隔离环境中连续实现、测试和修复                 | 低风险工具、内部原型、批量机械变更 | 权限最小化、预算和停止条件、最终审核 |
| L4 高风险操作 | 涉及生产、真实数据、支付、权限、固件或不可逆迁移 | 原则上不应无人值守                 | 强制双重审批、备份、演练和实时监控   |

## 25. Definition of Done：AI 任务完成标准
> **1 需求与验收标准已经满足，没有悄悄扩大范围。**
>
> **2 实现与批准的代码方案一致；偏差有说明并经过审核。**
>
> **3 构建、静态检查、相关自动化测试和真实流程验证通过。**
>
> **4 关键边界、异常、权限、并发和数据不变量有证据。**
>
> **5 日志、指标和错误信息足以诊断；没有泄露敏感数据。**
>
> **6 性能满足预算，或明确记录尚未满足的差距与风险。**
>
> **7 依赖、许可证、配置、迁移和兼容性已检查。**
>
> **8 发布、回滚和监控办法明确；高风险操作获得审批。**
>
> **9 文档、变更说明、ADR 和运行手册同步。**
>
> **10 本次新发现的问题已转成回归测试、规则或 Skill 更新。**

## 26. 如何衡量个人实践中估算的“约十倍效率”
建议把效率拆成速度、质量、维护、成本和学习五个维度，建立使用 AI 前后的基线。不要只比较写代码时间。

| **维度** | **推荐指标**                                                         |
|----------|----------------------------------------------------------------------|
| 速度     | 从需求确认到可用版本的 Lead Time；每周完成的可验收能力；首次通过率   |
| 质量     | 测试外逃缺陷、线上故障、回滚率、变更失败率、严重问题数               |
| 恢复     | 定位时间、修复时间、平均恢复时间、重复事故率                         |
| 维护     | 后续修改所需时间、复杂度、重复代码、依赖数量、文档新鲜度             |
| 成本     | 人工时间、模型调用、云资源、测试环境、第三方服务与长期维护           |
| 学习     | Skill 复用率、重复问题下降、跨项目迁移速度、新领域达到有效产出的时间 |
| 价值     | 用户采用、业务指标、实验结论、实际节省或新增收益                     |
