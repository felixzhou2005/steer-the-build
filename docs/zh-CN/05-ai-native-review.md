## 8.1 AI 原生代码 Review：Review 也必须规模化
当 AI 能在短时间内生成几十万行代码时，继续要求人逐行阅读、逐行修改，会让 Review 成为新的瓶颈。新的做法不是取消 Review，而是把 Review 的主要执行工作也交给 AI，让它系统化扫描整个代码库和变更集；人从“逐行编辑者”转为“审查目标设计者、风险裁决者和最终责任人”。

| **Review 层级** | **AI 应检查什么**                                                     | **主要证据**                                      | **人必须控制什么**                          |
|-----------------|-----------------------------------------------------------------------|---------------------------------------------------|---------------------------------------------|
| A. 架构与边界   | 模块职责、依赖方向、公共接口、数据所有权、架构约束和不变量            | 依赖图、ADR 对照、跨层调用清单、变更影响图        | 架构取舍、例外批准、长期演进                |
| B. 功能与需求   | 需求覆盖、状态流、边界条件、权限、异常、兼容与未实现项                | 需求追踪矩阵、场景测试、API 契约、差异清单        | 业务语义、验收优先级、风险接受              |
| C. 实现细节     | 并发、资源释放、SQL、事务、错误处理、安全、重复逻辑、复杂度和性能热点 | 静态分析、代码查询、基准、调用链、漏洞与依赖报告  | 高风险路径抽样、不可逆操作、关键算法        |
| D. 验证证据     | 构建、测试、覆盖、日志、迁移、回滚、监控和文档是否真实可用            | CI 结果、截图/Trace、测试报告、运行日志、演练记录 | Oracle 是否可信、证据是否充分、是否允许发布 |

> **•** Review AI 必须先读取需求、架构、项目 Skill 和变更计划，不能只看 diff。
>
> **•** 把“发现问题”和“修改代码”分离：先输出有证据的 Review 报告，经确认后再修复，避免一边审一边改变审查对象。
>
> **•** 对安全、权限、财务、迁移、并发和复杂算法使用独立上下文、第二模型或对抗角色，降低同源偏差。
>
> **•** 人不需要逐行阅读全部代码，但必须深入抽查高风险路径、审查架构变化，并对发布承担最终责任。

## 28. 一个建议的项目仓库“AI 宪法”
为了减少每轮对话重新解释，可以在仓库中维护一份 AI 可读取的项目说明，作为长期上下文。它不代替文档，而是索引和约束入口。

| **栏目**   | **内容**                                             |
|------------|------------------------------------------------------|
| 项目目标   | 用户、核心能力、成功指标和非目标                     |
| 架构地图   | 模块职责、依赖方向、关键数据流和高风险区域           |
| 开发规则   | 语言、风格、目录、错误处理、日志、测试和依赖政策     |
| 常用命令   | 安装、构建、测试、格式、基准、运行和调试             |
| 变更政策   | 允许自动修改的范围、必须审批的变更和禁止事项         |
| 质量门     | 哪些检查必须通过，哪些指标不能退化                   |
| 当前决策   | 活跃 ADR、已知限制、迁移状态和近期风险               |
| Skill 索引 | 按 SQL、UI、测试、性能、ML、硬件等场景加载对应 Skill |

## 28.1 一次 AI 原生 Review 的推荐工作流
> **1 冻结审查对象**：固定需求、架构、代码版本、配置和测试数据；审查期间不允许实现 Agent 静默继续修改。
>
> **2 建立追踪矩阵**：把需求、风险、文件、接口、测试和证据关联起来，先找未覆盖区域。
>
> **3 分层运行 Review Agent**：分别审查架构、功能、安全、数据、性能、可维护性和文档，避免一个超长提示包办所有问题。
>
> **4 运行真实工具**：构建、pytest、Playwright、Maestro、静态分析、迁移演练、基准和日志查询必须产生机器证据。
>
> **5 输出有证据的问题清单**：每个问题包含严重度、位置、触发场景、证据、影响和最小修复建议；不接受泛泛而谈。
>
> **6 由人做风险门**：决定哪些必须修、哪些接受、哪些需要重新设计，并批准高风险方案。
>
> **7 修复后独立复审**：修复 Agent 不负责最终证明；重新运行测试和独立 Review，检查是否引入新回归。
>
> **8 沉淀系统能力**：更新需求、架构、测试、日志、Review 规则和 Skill，而不只合并代码。
