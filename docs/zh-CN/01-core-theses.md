# 执行摘要：十五条核心结论
> **1 AI 编程的本质是工程分工重构**：AI 不只是提高敲代码速度，而是把人的价值上移到目标、需求、架构、约束、验证和复盘。
>
> **2 完整项目历程不能省略**：需求、设计、实现、测试、发布和复盘仍然都需要；AI 只会放大流程的优点，也会放大流程的缺陷。
>
> **3 架构决策必须由人负责**：AI 可以给方案、分析取舍、补全细节，但没有长期责任、完整业务上下文和真实风险承担能力。
>
> **4 先审方案，再让 AI 写代码**：每个实现任务都先形成代码方案、影响范围、接口、测试与风险，经审核后再执行，能显著提高可控性。
>
> **5 大需求必须拆成小而可验证的任务**：一个任务应当只回答一个主要工程问题，变更范围有限，结果可以独立验证。
>
> **6 测试、日志和指标构成主要信任机制**：大规模 AI 代码不再适合依靠人工逐行 Review，应当通过 AI 分层审查、契约、自动化测试、运行日志、指标和基准证明行为。
>
> **7 AI 代码需要专门的性能治理**：过度设计、冗余兜底、复杂 SQL、重复转换和不必要抽象是常见问题，必须用预算、剖析和基准来约束。
>
> **8 复杂算法的方向和评价体系由人控制**：AI 适合搜索、整理、比较和实现；算法目标、假设、基线、数据与折中必须由人确定。
>
> **9 Skill 是个人工程经验的可执行化**：把编码风格、约束、反模式、验证方法和例外条件固化为可版本化的 Skill，效率会持续复利。
>
> **10 结果不符时，反推控制面而不是只修补代码**：每次失败都应追问：是需求、架构、计划、约束、测试、日志还是 Skill 有缺陷，并把修复沉淀为系统能力。
>
> **11 Review 本身必须 AI 原生化**：让独立的 AI 审查架构边界、需求覆盖、权限安全、并发、SQL、性能和异常路径；人集中处理高风险取舍、关键抽样和责任签署。
>
> **12 文档是 AI 的控制面和外部记忆**：AI 通过文字理解系统，人的思考过程必须被规格化、版本化和持续更新；文档缺失会直接转化为上下文漂移和返工。
>
> **13 可观察性必须同时服务人和 AI**：结构化日志、追踪、指标、模型与数据版本以及诊断地图，让 AI 能够检索真正相关的运行证据并定位第一个异常点。
>
> **14 人的价值上移而不是消失**：审美与主题、非通用架构、复杂算法、技术选型、性能数据建模和异常拆解仍需要人的经验、直觉和责任。
>
> **15 AI 会扩大软件的应用面**：开发成本下降后，过去因 ROI 不足而被取消的个性化系统会变得可行；同时必须用平台化、治理和可维护性避免客户定制碎片化。

## 15. 十五条可直接执行的原则
| **编号** | **原则**               | **实践含义**                                                   |
|----------|------------------------|----------------------------------------------------------------|
| P1       | 文档先于代码           | 任何重要功能都先形成可验证规格。                               |
| P2       | 架构责任不委托         | AI 提供选项，人做取舍并记录理由。                              |
| P3       | 方案先于实现           | 先审文件、接口、数据流、测试和风险，再生成代码。               |
| P4       | 小批次、单关注点       | 每个任务独立验证、独立解释、尽量独立回滚。                     |
| P5       | 默认简单               | 不为未知未来增加抽象、兜底和通用性。                           |
| P6       | 接口和不变量优先       | Review 先看边界、状态、数据和失败行为。                        |
| P7       | 证据优先于自信         | 运行结果、测试、日志、指标和基准比模型解释更可信。             |
| P8       | 测试需要对抗视角       | 避免生成者用同一假设为自己证明。                               |
| P9       | 从第一天建设可观测性   | 让系统可还原、可定位、可比较。                                 |
| P10      | 先测量再优化           | 性能改善必须有基线和回归保护。                                 |
| P11      | 高风险操作提高审批门槛 | 生产、数据、安全、支付、权限和不可逆变更不应自动放行。         |
| P12      | 失败要反推流程         | 修代码之外，还要修规格、测试、日志或 Skill。                   |
| P13      | Skill 必须可维护       | 有范围、版本、例外、验证和过期机制。                           |
| P14      | 研究必须回到实验       | 资料整理和模型建议只能形成假设，不能代替基线与实证。           |
| P15      | 效率看全周期           | 不以生成速度或代码行数衡量，而看交付、缺陷、维护、成本和学习。 |
