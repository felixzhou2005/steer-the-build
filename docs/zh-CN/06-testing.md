## 9. 阶段 6：测试是 AI 代码的核心信任机制
即使前面的流程严格，AI 生成的系统仍然会有大量问题。自动化测试不是可选项，而是规模化委托编码和审查之后的基础设施。AI 可以生成并驱动测试，但“让同一个 AI 用同一套假设证明自己正确”存在同源偏差，因此测试目标、关键断言和业务 Oracle 必须来自需求，重要场景应由独立审查代理或不同上下文进行对抗验证。

### 建议的证据组合
| **证据类型**       | **主要价值**                     | **特别适合发现**                         |
|--------------------|----------------------------------|------------------------------------------|
| 单元测试           | 快速验证局部规则和边界           | 分支、异常、计算和状态转换错误           |
| 契约测试           | 稳定模块、API、协议和数据边界    | 接口漂移、兼容性和序列化问题             |
| 集成测试           | 验证真实组件协作                 | 数据库、网络、队列、文件和第三方依赖问题 |
| 端到端/UI 流程测试 | 验证用户可见的完整旅程           | 页面状态、导航、权限和多步骤流程问题     |
| 属性/不变量测试    | 验证一类输入都必须成立的规则     | AI 未枚举到的边界与组合错误              |
| 差分/基准对照      | 与旧实现、简单基线或参考算法比较 | 复杂算法、性能和数值偏差                 |
| 回归测试           | 把每次事故永久固化为证据         | 重复出现的已知故障                       |
| 静态与安全扫描     | 在运行前发现结构和已知风险       | 类型、依赖、秘密、漏洞和不安全模式       |

覆盖率只能说明“执行过哪些代码”，不能说明断言是否正确。更有价值的问题是：测试是否覆盖业务不变量、失败路径、竞争条件、迁移、回滚和真实用户流程；测试能否在实现被故意破坏时失败。

## 9.1 测试工具是大规模 AI 代码的主要 Review 基础设施
AI 代码量大、变化快，质量不能主要依赖人眼阅读。测试工具提供稳定、可重复、可自动执行的行为证据，并允许 AI 生成用例、运行测试、分析失败和补充回归。关键不是“用了某一个测试框架”，而是建立从函数到真实用户流程、从正常路径到故障路径的证据金字塔。

| **工具 / 层级**          | **典型用途**                                              | **AI 最适合承担的工作**                          | **必须防止的问题**                         |
|--------------------------|-----------------------------------------------------------|--------------------------------------------------|--------------------------------------------|
| pytest                   | Python 单元、集成、属性、算法回归与数据校验               | 从规格生成测试、参数化边界、运行与最小化失败样本 | 按实现抄测试、断言过弱、Mock 过度          |
| Playwright               | Web 端到端、跨浏览器、API/UI 流程、截图、Trace 和网络证据 | 生成真实流程、抓取失败工件、分析 DOM/网络/控制台 | 只测 happy path、选择器脆弱、环境不真实    |
| Maestro                  | iOS / Android / Web 的黑盒 UI 与端到端用户旅程            | 用可读 Flow 驱动模拟器/设备、复现权限和状态流程  | 设备差异、时间等待、系统弹窗和测试数据污染 |
| 静态分析 / 类型 / Lint   | 类型、死代码、复杂度、依赖、常见安全缺陷和风格一致性      | 批量扫描、归类问题、提出最小修复                 | 把警告全部压制或为通过规则过度重构         |
| 契约 / 数据库测试        | API Schema、兼容性、迁移、事务、索引、数据不变量          | 生成契约样例、迁移演练、差分和回滚验证           | 只验证脚本可运行而不验证数据语义           |
| 差分 / Golden / 属性测试 | 复杂算法、解析器、模型特征、序列化和兼容性                | 构造随机与边界输入、比较参考实现和新实现         | 错误参考实现成为唯一真值                   |
| 性能与稳定性测试         | 延迟、吞吐、内存、耗电、包体、长稳、并发和故障恢复        | 自动跑基准、定位回归、比较版本                   | 无基线、环境漂移、用平均值掩盖尾延迟       |
| 硬件在环 / 故障注入      | IoT、嵌入式、网络中断、掉电、传感器异常和设备状态机       | 编排场景、采集日志、对比预期状态                 | 软件模拟替代物理约束、安全边界不清         |

> **•** 测试从需求和风险派生，不从当前实现反向抄写；关键业务断言必须由人确认。
>
> **•** 生成代码的 Agent、生成测试的 Agent、执行与审查证据的 Agent 应尽量分离上下文。
>
> **•** 每次失败保留可复现输入、环境、版本、日志、截图、视频、Trace 和最小失败用例。
>
> **•** 测试套件本身也要 Review：检查是否遗漏状态、异常、权限、竞争条件和恢复路径。
