## 3. 阶段 0：先定义问题，而不是先生成代码
原始体会从需求文档开始，这已经比“想到一个功能就让 AI 实现”成熟很多。进一步补充的一层是：在需求之前先验证问题和价值。AI 能把错误的想法实现得非常快，因此更需要先确认方向。

### 应当明确的内容
> **•** 目标用户、使用场景和当前痛点；这个问题为什么值得解决。
>
> **•** 业务目标、技术目标和学习目标分别是什么，三者不要混在一起。
>
> **•** 成功指标与失败边界；哪些结果即使“功能完成”也不能接受。
>
> **•** 替代方案，包括不开发、人工流程、购买现成方案或只做实验。
>
> **•** 范围之外的内容，以及何时才会重新考虑它们。

## 4. 阶段 1：需求必须先文档化
需求分析不能在对话中边想边实现。需求散落在多轮聊天里，会导致范围漂移、上下文丢失、局部补丁相互冲突，最后形成无法解释的系统。详细需求文档的价值不是形式，而是建立统一、稳定、可回溯的事实源。

### 一份可执行的需求规格至少包括
> **•** 问题陈述、用户角色、核心场景与端到端用户旅程。
>
> **•** 功能规则、业务不变量、状态变化和数据生命周期。
>
> **•** 正常路径、异常路径、边界条件、权限与并发情形。
>
> **•** 可验证的验收标准，尽量使用示例和反例，而不是模糊形容词。
>
> **•** 范围内、范围外、依赖项、假设和待确认问题。
>
> **•** 需求变更记录：改了什么、为什么改、影响哪些设计与测试。

最有效的要求之一，是让 AI 在写代码前先把需求转成“场景—输入—预期结果—异常—证据”的结构。如果这一步说不清，编码通常只会把歧义固化。

## 4.1 文档不是副产品，而是 AI 的控制面和外部记忆
传统项目中，文档经常被看成实现之后的总结；在 AI 编程中，文档本身就是输入质量、上下文连续性和工程控制力。AI 只能根据可见文字、代码和工具证据推导，人的隐含意图如果没有被外部化，就会在长对话、上下文切换和多轮修改中丢失。

详细文档看起来增加了前期工作，却能显著减少返工。它同时完成三件事：迫使人把思路系统化；为 AI 提供稳定、可复用的上下文；为后续 Review、测试、升级和故障排查留下可追溯依据。

| **文档链条**     | **必须回答的问题**                               | **与下一阶段的关系**        |
|------------------|--------------------------------------------------|-----------------------------|
| 问题与价值说明   | 为谁解决什么问题，为什么值得做，什么不做         | 约束 PRD 和投资边界         |
| PRD / 需求规格   | 场景、规则、状态、异常、边界、验收标准           | 驱动设计、测试和业务 Review |
| 非功能与风险规格 | 性能、安全、隐私、可靠性、成本、兼容与运维       | 驱动架构和质量门            |
| UX / Figma 规格  | 布局、主题、组件、状态、交互、响应式与可访问性   | 驱动 UI 实现和视觉回归      |
| 架构与 ADR       | 模块边界、数据流、依赖、权衡、演进和回滚         | 驱动任务拆分和架构 Review   |
| API 与数据设计   | 契约、错误、幂等、权限、表与字段语义、索引和迁移 | 驱动实现、集成和数据测试    |
| 算法与实验规格   | 假设、基线、目标函数、指标、数据、误差和停止条件 | 驱动可复现实验与算法验收    |
| 实现计划         | 影响范围、步骤、风险、测试、文档和回滚           | 控制单次代码变更            |
| 测试与验收计划   | 测试层级、Oracle、覆盖、环境、数据和证据产物     | 决定什么叫完成              |
| 发布与运维手册   | 部署、配置、监控、告警、灰度、回滚和事故处理     | 使系统可运行、可恢复        |
| 变更记录与复盘   | 发生了什么、为什么、证据、根因和 Skill 更新      | 把经验写回控制面            |

## 5. 阶段 2：非功能需求和风险必须前置
很多性能、安全和运维问题并不是代码完成后才出现，而是因为一开始没有给出约束。AI 在没有约束时会选择“看起来完整”的方案，这往往意味着额外抽象、额外兜底和额外成本。

| **维度**   | **需要提前回答的问题**                                                 |
|------------|------------------------------------------------------------------------|
| 性能       | 响应时间、吞吐量、并发、内存、包体、启动时间、耗电或实时性预算是什么？ |
| 可靠性     | 允许怎样失败？是否需要重试、幂等、降级、恢复、备份和灾难恢复？         |
| 安全与隐私 | 威胁模型、权限边界、敏感数据、密钥、审计和法规要求是什么？             |
| 兼容性     | 支持哪些平台、版本、协议、设备、数据库和历史数据？                     |
| 可维护性   | 模块边界、复杂度上限、依赖策略、文档和测试要求是什么？                 |
| 可运维性   | 如何监控、告警、追踪、发布、灰度、回滚和处理事故？                     |
| 成本       | 模型调用、云资源、存储、带宽、第三方服务和维护成本上限是什么？         |
