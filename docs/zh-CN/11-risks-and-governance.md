## 18. 需要补齐的 P0 级维度：应尽快纳入方法论
### 18.1 产品价值与需求验证
当前体会强调需求文档，但还可以再向上追问：需求是否值得做、是否有用户证据、是否存在更低成本的替代方案。AI 降低实现成本后，最容易出现的是“开发了很多正确实现的错误功能”。

> **•** 建议增加“问题说明书”：目标用户、痛点证据、成功指标、替代方案、停止条件。
>
> **•** 在完整实现前，用原型、假数据、人工流程或最小实验验证关键假设。

### 18.2 安全、隐私与合规
安全不能只靠测试末尾扫描。AI 可能复制不安全模式、泄露密钥、错误处理权限、记录敏感数据或引入高风险依赖。业务、Mobile、ML 和硬件项目都可能涉及真实用户、设备和数据。

> **•** 在设计阶段做简化威胁建模：资产、攻击者、入口、权限边界、滥用方式和缓解措施。
>
> **•** 对认证、授权、支付、密钥、加密、上传、反序列化、命令执行和生产数据设置强制人工审查。
>
> **•** 日志默认脱敏；敏感数据最小化收集；明确保留、删除和访问审计。

### 18.3 发布、迁移、回滚和事故处理
“代码能运行”与“产品可安全上线”之间还有很长距离。数据库迁移、配置、模型版本、设备固件、App 商店发布和第三方接口变化都可能产生不可逆后果。

> **•** 每个高风险变更都要有发布计划、观测窗口、回滚条件和负责人。
>
> **•** 数据迁移采用扩展—迁移—收缩等分阶段策略，避免代码和数据瞬间强耦合。
>
> **•** 建立事故复盘模板，把线上问题转成测试、监控、规则和 Skill。

### 18.4 依赖、供应链与许可证
AI 很容易为了完成任务引入新库、复制常见代码或选择不再维护的依赖。依赖的安全、许可证、体积、平台兼容和长期维护成本需要单独治理。

> **•** 默认禁止无理由新增依赖；新增时说明必要性、替代方案、许可证、维护状态和升级策略。
>
> **•** 锁定版本、生成软件物料清单、扫描漏洞，并避免把模型输出的来源不明代码直接用于敏感项目。

### 18.5 测试的独立性与真实性
AI 能快速生成大量测试，但测试可能只是复述同一实现逻辑，或者为了通过而降低断言。测试数量多不等于证据强。

> **•** 先由需求、不变量和历史事故定义测试，再让 AI 实现测试。
>
> **•** 对关键算法使用简单基线、独立实现、真实样本和差分比较。
>
> **•** 让“对抗审查者”专门尝试破坏实现，检查测试是否会真正失败。

### 18.6 变更控制与版本管理
AI 的高吞吐会放大并行修改、范围外改动和合并冲突。需要明确仓库级规则，而不仅是单次提示。

> **•** 使用小分支、小提交、清晰变更说明和自动门禁；禁止 AI 悄悄修改无关文件。
>
> **•** 重大架构、数据、依赖和公共接口变更必须有 ADR 或等价记录。
>
> **•** 任何自动生成或格式化都应可重复，避免产生无法审查的噪声差异。

### 18.7 可维护性、知识所有权与能力退化
完全不亲手编码在个人场景中可以成立，但长期仍要避免系统成为只有 AI 能修改的黑箱。负责人需要维护模块地图、关键决策和故障知识，也要防止自身代码阅读和调试能力退化。

> **•** 要求每个模块有职责、接口、不变量、数据流和运维说明。
>
> **•** 定期让 AI 生成“从入口到结果”的代码路径解释，再由人抽样核对。
>
> **•** 对关键模块进行风险导向的人工阅读，而不是平均逐行阅读全库。

### 18.8 端到端效率与经济指标
“效率提升十倍”非常有价值，但最好建立可重复的衡量方式。代码行数不适合作为主要指标，因为 AI 既可能提高产出，也可能制造膨胀。

> **•** 比较从想法到可用版本的周期、人工投入、首次通过率、返工、线上缺陷、恢复时间和维护成本。
>
> **•** 记录模型调用、云资源、测试环境、第三方服务和未来维护的总成本。
>
> **•** 区分“做得更多”和“单位价值更高”，避免吞吐增长掩盖产品价值下降。

## 19. 需要补齐的 P1 级维度：形成长期工程体系
| **维度**             | **建议**                                                                                       |
|----------------------|------------------------------------------------------------------------------------------------|
| 上下文工程与仓库说明 | 建立项目宪法、目录地图、编码规则、常用命令、禁改区域和当前决策，减少每轮重新解释与上下文漂移。 |
| 文档同步             | 代码、API、数据库、配置和运行手册变化后自动检查文档是否过期，避免文档只在项目开始时准确。      |
| Skill 治理           | 版本化、评审、测试、设定维护者和失效条件，处理多个 Skill 之间的优先级冲突。                    |
| 团队协作与角色       | 明确产品、架构、AI 操作、测试、安全和上线审批的责任，不让“AI 写的”成为无人负责的理由。         |
| 成本与工具依赖       | 监控模型成本、速度、可用性和供应商变化；保留可切换的文档、测试和标准接口。                     |
| 可访问性与国际化     | UI 设计阶段纳入键盘、读屏、对比度、动态字体、多语言、时区和本地格式。                          |
| 并行代理与冲突控制   | 多个 AI 同时工作时按模块和任务分区，定义合并顺序和共享资源，避免互相覆盖。                     |
| 事故演练与韧性       | 定期演练依赖不可用、数据损坏、模型异常、设备断连和发布失败，而不是只等真实事故。               |

## 20. 原始结论中值得进一步校准的几个观点
| **原始观点**               | **建议的精确表达**                                                                                                             |
|----------------------------|--------------------------------------------------------------------------------------------------------------------------------|
| “完全不需要自己写一行代码” | 可以作为你的个人实践结论，但对外表达时最好加上条件：项目负责人仍需具备足够的架构、测试和诊断能力，并承担全部工程责任。         |
| “没必要逐行读代码”         | 对大规模生成代码是现实判断，但应补充风险导向 Review：安全、权限、数据迁移、并发、公共接口、算法核心和不可逆路径需要深入检查。  |
| “AI 会过度设计”            | 这个现象常见，但根因往往包括需求模糊、非功能约束缺失、提示鼓励“完整方案”以及没有简单性门槛。解决方法是把简单性写成可验证约束。 |
| “复杂算法一定自己先研究好” | 方向正确。还应增加基线、数据切分、误差分析、消融实验和可复现记录，防止人和 AI 一起被漂亮但错误的解释说服。                     |
| “从 0 到 1 不必先系统学习” | 问题驱动学习非常高效，但阶段性需要回头系统整理概念图、关键假设和知识空白，否则容易形成碎片化理解与确认偏差。                   |
| “多模型可以互相审核”       | 有帮助，但多个模型可能共享相似训练模式和盲点。真正独立的证据仍然是源资料、运行实验、参考实现和真实用户结果。                   |
| “30 万行代码”              | 能说明 AI 的实现吞吐，但不代表系统价值或质量。更值得跟踪的是功能密度、复杂度、重复率、变更成本和缺陷逃逸率。                   |

## 21. 常见反模式与修正办法
| **反模式**             | **典型后果**                             | **修正**                                   |
|------------------------|------------------------------------------|--------------------------------------------|
| 想到哪里做到哪里       | 功能能快速出现，但需求互相冲突、返工不断 | 先维护唯一需求源；变更先更新规格和影响分析 |
| 一个大提示完成整个系统 | 变更巨大、上下文丢失、无法隔离问题       | 按垂直能力和关注点拆分，小批验证           |
| 让 AI 自由决定架构     | 过度抽象、错误折中、性能和运维问题       | 人给约束；AI 出选项；ADR 记录人工取舍      |
| 代码能跑就验收         | 边界、异常、安全和兼容问题留到生产       | 定义多层证据和质量门                       |
| AI 写代码又写同源测试  | 测试只是复述实现，错误假设被共同固化     | 从需求和不变量出发，加入独立对抗测试       |
| 出问题就继续打补丁     | 兜底越来越多，根因被遮蔽                 | 先复现、观测、隔离和解释因果，再修复       |
| 为了保险过度兜底       | 复杂度、延迟和不可预测行为增加           | 只处理规格内失败；额外容错需有明确故障模型 |
| 全量逐行 Review        | 人力不可扩展，仍可能错过系统级问题       | 风险分级，优先审边界、不变量和高危路径     |
| Skill 只增不减         | 规则冲突、陈旧经验压制新方案             | 版本、测试、失效条件和定期清理             |
| 用代码行数证明效率     | 鼓励膨胀，掩盖维护成本                   | 用端到端交付、质量、成本和业务结果衡量     |
