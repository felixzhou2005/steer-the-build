## 11. 阶段 8：性能必须按证据治理
AI 代码常见的问题不是“不会实现”，而是实现得过多、过深或过于通用。复杂 SQL、冗余转换、重复 I/O、过度缓存、层层包装和不必要的防御逻辑都会在规模放大后暴露。性能优化不能靠感觉，也不能只要求“写得快”，而要从预算和测量开始。

### 性能治理闭环
> **•** 先定义预算：响应时间、吞吐量、内存、CPU、网络、耗电、包体或实时性。
>
> **•** 建立可重复基准，保留输入、数据量、环境和版本。
>
> **•** 使用 Profiler、查询计划、火焰图、追踪和系统指标定位热点。
>
> **•** 一次只改变一个主要因素，并与基线比较。
>
> **•** 优化后补充性能回归测试或阈值监控。
>
> **•** 对 SQL 明确要求简单、可解释、索引友好，并用执行计划验证，而不是只看语句表面。

“简单 SQL”不等于“短 SQL”。真正的标准是：语义正确、执行计划可控、数据规模下稳定、锁和事务边界清楚。

## 12. 阶段 8 的另一半：发布、回滚与运行
原始体会重点集中在开发阶段，生产交付是最需要补齐的环节之一。AI 可以执行构建、部署、迁移和运维命令，但越接近生产、数据和不可逆操作，越必须提高人工审批和证据门槛。

| **环节**   | **最低要求**                                                 |
|------------|--------------------------------------------------------------|
| 持续集成   | 每次变更自动构建、测试、扫描；失败时禁止合并。               |
| 版本与变更 | 小批提交、可追溯变更说明、发布说明和兼容性标记。             |
| 数据迁移   | 向前/向后兼容、分阶段迁移、备份、校验和回滚或补偿方案。      |
| 发布控制   | 功能开关、灰度或分批发布；高风险能力默认关闭。               |
| 监控告警   | 围绕用户结果、错误率、延迟、资源、队列和关键业务不变量设置。 |
| 回滚恢复   | 代码、配置、模型和数据都要考虑恢复路径，并定期演练。         |
| 事故响应   | 明确停止扩散、保存证据、恢复服务、根因分析和后续行动。       |

## 20.1 AI DevOps：开发测试环境高度自动化，生产环境受控执行
AI 已经能够很好地理解 Linux、编写脚本、安装依赖、分析服务、修改配置、执行部署和排查问题。在开发与测试环境中，让 AI 直接操作命令行能显著减少运维摩擦，尤其适合底层、跨服务和部署复杂的项目。但生产环境不能把“能力强”误解为“可以无限授权”。

| **环境**         | **建议的 AI 权限**                                                   | **必须的控制**                                               |
|------------------|----------------------------------------------------------------------|--------------------------------------------------------------|
| 本地开发 / 沙箱  | 可高自治执行安装、构建、测试、容器、日志查询和环境重建               | 隔离、可删除、密钥最小化、命令审计                           |
| 共享测试环境     | 可部署、迁移测试数据、运行端到端和故障注入                           | 资源配额、测试租户、并发锁、可恢复快照                       |
| 预发布 / Staging | 执行与生产同源的已版本化脚本，验证监控、灰度和回滚                   | 审批、dry-run、差异报告、演练记录                            |
| 生产环境         | 原则上只执行经过验证、版本化、最小权限的操作；高风险步骤要求人工批准 | 双人或变更审批、备份、金丝雀、回滚、审计、紧急停止和凭据隔离 |
