## 13. 阶段 9：把经验沉淀成 Skill
Skill 的价值不只是节省重复提示词，而是把个人判断、项目约束和质量门转成可执行的工程政策。Skill 应分层管理，并明确适用范围、正反例、验证方式、版本、冲突关系和例外升级路径。

### 一个完整 Skill 应包含
> **•** 适用范围与不适用范围；触发条件和前置输入。
>
> **•** 必须遵守的原则、禁止模式和优先级冲突规则。
>
> **•** 期望输出结构、步骤、检查点和停止条件。
>
> **•** 正例、反例以及常见失败模式。
>
> **•** 验证命令、测试、基准或人工审批要求。
>
> **•** 版本、变更原因、维护者和过期条件。

## 13.1 Skill 分层：从底层原则到项目质量门
单一、庞大的 Skill 很容易互相冲突，也难以在不同项目复用。更有效的方式是分层：底层原则尽量稳定，上层 Skill 越接近具体项目和具体任务。执行一次任务时，只加载真正相关的层级。

| **层级**               | **作用**                   | **典型内容**                                                   | **示例**                                                 |
|------------------------|----------------------------|----------------------------------------------------------------|----------------------------------------------------------|
| L0 基础宪法 / 原则     | 所有项目默认遵守的底线     | 先方案后实现、简单优先、不扩大范围、证据完成、不可逆操作升级   | 任何新增抽象、依赖、兜底都必须证明必要性                 |
| L1 领域 / 技术栈 Skill | 约束某类技术的惯用做法     | SQL、API、iOS、Web、ML、语音、IoT、安全、DevOps                | SQL 以简单高效为默认，先查执行计划，不做无需求的通用化   |
| L2 项目 Skill          | 把具体项目的架构和语义固化 | 模块边界、命名、API、字段语义、权限、Figma、日志字典、部署约束 | UI 基于指定 Figma 高保真实现，不允许 AI 自由改主题或结构 |
| L3 任务 / 质量门 Skill | 控制某一次高风险操作       | 迁移、发布、性能、事故、视觉验收、模型上线、硬件升级           | 迁移必须 dry-run、备份、数据对账、回滚演练和人工批准     |

## 13.2 从缺陷到 Skill：把每次问题转成长期能力
> **1 记录现象与证据**：保存需求、代码、日志、测试、实际影响和当时的 AI 行为。
>
> **2 判断缺的是哪类控制**：需求、架构、计划、测试、日志、工具、权限还是已有 Skill 的表述。
>
> **3 写最小规则**：只约束真正导致问题的条件，避免用一个事故制造过度通用的禁令。
>
> **4 加入正例与反例**：让 AI 看懂什么是合格结果，什么是常见但错误的做法。
>
> **5 用历史失败验证**：重新运行旧场景，确认 Skill 能预防或更早发现问题。
>
> **6 版本化与退役**：记录所有者、适用项目、依赖、冲突、模型兼容和废弃条件。

## 14. 工具分工：按工作阶段选工具，而不是争论品牌
根据原始实践，ChatGPT Pro 更适合复杂算法研究、思路推演和关键代码雏形；Codex 更适合结合本地代码库进行方案细化、实现、测试和运行分析；Claude 作为偶尔使用的补充工具。这里最重要的不是给工具排永久名次，而是建立阶段化分工。

| **工作类型**         | **推荐角色分工**                                                                   |
|----------------------|------------------------------------------------------------------------------------|
| 从 0 到 1 的问题研究 | 让通用对话模型帮助铺开资料、概念、方案空间和实验方向；人负责验证资料、建立判断。   |
| 复杂算法设计         | 人先明确目标、约束、基线与评价；模型协助比较方法、推导细节和生成原型。             |
| 本地代码库实现       | 让代码代理读取真实仓库、生成代码方案、执行受限变更、运行测试并解释差异。           |
| 关键决策复核         | 可以用另一个模型从不同提示和角色进行反驳式审查，但不能把多模型一致误认为事实证明。 |
| 问题诊断             | 让模型基于日志、调用链、代码路径和可重复实验建立证据，而不是凭现象猜根因。         |

## 28.2 Skill 治理：数量多以后，必须像代码一样管理
> **•** 每个 Skill 有唯一名称、层级、所有者、版本、适用项目、依赖、冲突和废弃状态。
>
> **•** 为 Skill 建立最小测试集：至少一个正例、一个反例、一个历史失败和一个不应触发的场景。
>
> **•** 模型或工具升级后执行回归，防止旧 Skill 变得过强、失效或与新能力重复。
>
> **•** 项目 Skill 只能收录长期约束，临时任务细节留在任务方案，避免上下文持续膨胀。
>
> **•** 冲突时按层级和明确性处理：项目特殊约束可以覆盖通用 Skill，但必须记录原因和退出条件。
