# 为 Steer the Build 做贡献

[English](CONTRIBUTING.md)

Steer the Build 欢迎真实经验、反例、模板、Skill、翻译和纠错。这个项目以证据为导向：读者能够看到做过什么、发生了什么、如何衡量，以及结论在哪些条件下不再成立时，贡献才真正有价值。

## 提交前先选择合适的渠道

| 渠道 | 适用内容 |
|---|---|
| GitHub Discussions | 早期想法、开放问题、经验交流和方法争论 |
| Issues | 边界清楚的错误、内容缺口、已接受的案例提议或具体任务 |
| Pull Requests | 范围明确、可以审查并有证据支持的修改 |
| 私密安全报告 | 密钥、客户数据、危险运维指令或可被利用的示例 |

分享日志、Prompt、录音、基础设施细节或客户案例前，请先阅读 [SECURITY.md](SECURITY.md)。

## 优先欢迎的贡献

- 展示从意图到运行证据完整路径的案例；
- 一个根因最终改变了文档、测试、质量门或 Skill 的失败案例；
- 能够缩小、修正或推翻文档中某个结论的反例；
- 已经在真实项目使用过的模板；
- 具有明确触发条件、非触发条件、退出标准和验证历史的 Skill；
- 由熟悉技术语境的人审阅，并且绑定源 commit 的翻译；
- 对相关项目做准确、尊重事实的比较，说明重叠和差异。

## 证据等级

案例和贡献中的事实性结论应标明成熟度：

| 等级 | 含义 |
|---|---|
| `E0 — 假设` | 有理由支持，但尚未经过测试的想法 |
| `E1 — 单次观察` | 在一个项目或事故中观察到，且记录了背景 |
| `E2 — 重复观察` | 在明确条件下，多个任务或项目中重复出现 |
| `E3 — 可度量` | 有指标、测试工件、Trace 或基准支持 |
| `E4 — 独立审查` | 被原作者之外的人复现、质疑或审查 |

个人实践完全可以成为证据，但必须清楚标为个人经验，不能把一次观察直接写成普遍规律。

## 案例要求

使用 [`templates/zh-CN/case-study.md`](templates/zh-CN/case-study.md)。可发布的案例至少应说明：

1. 问题、用户、范围和成功标准；
2. 实现前知道什么，过程中哪些判断发生了变化；
3. 哪些决定由人负责，哪些工作交给 AI；
4. 架构、数据、UI、算法或运维约束；
5. 测试、运行证据，以及测试 Oracle 的来源；
6. 至少一个失败、意外、折中或限制；
7. 根因定位路径，尽量找到第一个可观察偏差点；
8. 事后新增了哪些文档、回归测试或 Skill；
9. 哪些事实没有公开以及原因；
10. 每个主要结论的证据等级。

不能为了让案例显得完整而编造指标、客户结果、引语、截图或架构细节。请直接使用 `TODO`、`未测量` 或 `因保密不公开`。

## Skill 要求

使用 [`templates/zh-CN/skill-spec.md`](templates/zh-CN/skill-spec.md)。一个 Skill 应包括：

- 边界清楚的目的；
- 正向触发条件和明确的非触发条件；
- 所需输入和前提；
- 可执行步骤；
- 约束和禁止的捷径；
- 预期输出和证据；
- 退出标准；
- 促使它产生的案例或历史失败；
- Owner、版本和复审日期。

文字看起来严格，不代表 Skill 有效。需要说明它怎样在真实任务或历史错误上被验证。

## 翻译规则

中文和英文是地位相同的两个版本，不要求逐段机械对应。

- 必须保留结论、置信程度、风险边界和证据等级；
- 优先使用自然的技术语言，而不是逐字翻译；
- 翻译 PR 应关联所采用的源 commit 或源文件版本；
- 不能在翻译中悄悄强化或弱化原结论；
- 术语没有稳定译法时，第一次保留原文并做简短解释。

详见 [`meta/translation-guide.md`](meta/translation-guide.md)。

## 使用 AI 辅助贡献

欢迎使用 AI，但贡献者仍然对准确性、许可证、隐私、引用和最终文字负责。

当 AI 实质参与了研究、翻译、案例还原或运维指令生成时，请在 Pull Request 中简短说明，例如：

> AI 辅助说明：用于重构初稿和检查双语一致性；所有事实与资料均由贡献者复核。

AI 的回答本身不能作为证据。请引用底层资料、实验、测试工件，或明确标注为个人观察。

## Pull Request 流程

1. 一个 PR 只处理一个案例、Skill、章节、模板或一组高度相关的修正；
2. 修改稳定核心结论时，应同步更新两个语言版本；暂时无法同步时，在 [`meta/content-status.md`](meta/content-status.md) 标为待翻译；
3. 按 [`meta/release-checklist.md`](meta/release-checklist.md) 运行文档检查；
4. 完成 PR Checklist，说明证据等级、来源和脱敏内容；
5. 面对 Review 意见，要修改结论、证据或适用范围，不能只把不确定内容写得更自信。

## 写作风格

- 从工程问题出发，不从工具品牌出发；
- 明确区分观察、推断、建议和预测；
- 版本、时间、环境和项目条件重要时要写准确；
- 用具体失败路径和工件替代口号；
- 主动写出边界和反例；
- 工具相关内容要标版本和日期，因为它比原则更容易过时；
- 不得公开密钥、个人信息、客户标识、私密录音、专有代码或可能影响安全的基础设施细节。
