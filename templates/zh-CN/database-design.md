Schema 和索引必须依据真实访问模式，而不是只依据通用范式或通用 NoSQL 建议。

## 3. 实体与表设计

每个表/Collection 应说明：

- 业务含义；
- Owner；
- 主键和标识语义；
- 字段类型、单位、Null、默认值和语义注释；
- 约束和不变量；
- 生命周期和删除行为；
- 预计规模和增长；
- 敏感字段；
- 生成或冗余字段以及更新规则。

## 4. 关系与一致性

- 事务边界：
- 引用完整性：
- 跨服务或跨存储一致性：
- Event/Outbox 策略：
- 冲突和重复处理：
- Read Model / 宽表 / 冗余理由：

冗余是有意识的性能取舍。必须说明冗余值怎样保持正确、怎样发现漂移。

## 10. Review Checklist

- [ ] 每个字段都有稳定业务含义。
- [ ] 重要不变量由约束保护。
- [ ] 关键访问模式有 Query Plan 证据。
- [ ] 冗余具有一致性机制。
- [ ] 迁移和回滚已经测试。
- [ ] 隐私、留存和删除明确。
- [ ] 可观察性能够定位慢操作或失败操作。
